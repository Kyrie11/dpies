# nuPlan v1.1 official-devkit integration changes

This revision extends the optimized DPIES-nuPlan code with official nuPlan v1.1 devkit integration for map-data extraction and closed-loop simulation.

## Added/updated map-data extraction

- Added `dpies/data/scenario_api.py` for official `NuPlanScenario` access.
  - Extracts route roadblock ids with `get_route_roadblock_ids()`.
  - Extracts current traffic-light states with `get_traffic_light_status_at_iteration(0)`.
  - Extracts future traffic-light state history with `get_future_traffic_light_status_history(...)` for teacher labels only.
  - Exposes `scenario.map_api` for exact HD-map geometry extraction.
- Reworked `dpies/data/map_provider.py`.
  - Supports `extract_from_api(map_api, ...)` for Scenario API and closed-loop `PlannerInitialization.map_api`.
  - Preserves structured local-frame geometry for route corridors, traffic-light connectors, stop lines, crosswalks, drivable areas, lane boundaries and speed-limit units.
  - Emits route and traffic-light metadata into `route_info_json`.
- Updated preprocessing (`dpies/data/preprocess_nuplan.py`).
  - New `--use-scenario-api` mode.
  - New `--map-version` and `--sensor-root` options.
  - Input GeometryQuery uses current traffic-light state only.
  - Teacher GeometryQuery uses future traffic-light status labels.

## Added exact map-rule GeometryQuery

- `dpies/evidence/geometry_query.py` now uses structured Shapely polygons/lines when available for:
  - drivable-area containment / outside area,
  - lane-boundary crossing,
  - stop-line crossing,
  - crosswalk intersection,
  - red traffic-light connector violation,
  - route deviation.
- Future traffic-light data is explicitly gated behind `use_future_traffic=True` so it is used for teacher labels, not model inputs.

## Added official closed-loop planner adapter

- Added `dpies/evaluation/closed_loop_planner.py` implementation of a nuPlan `AbstractPlanner`.
- The planner consumes official `PlannerInitialization.route_roadblock_ids`, `PlannerInitialization.map_api`, and per-step `PlannerInput.traffic_light_data`.
- Each simulation step builds DPIES batch tensors online, runs the trained model, performs capped greedy selection, chooses the max-min action, and returns a global `InterpolatedTrajectory`.
- Fallback stop rollout is available with `fallback_on_error=true`.

## Added closed-loop configs/scripts

- Added `configs/simulation/planner/dpies_planner.yaml`.
- Added:
  - `scripts/run_closed_loop_nonreactive.sh`
  - `scripts/run_closed_loop_reactive_idm.sh`
  - compatibility wrappers `scripts/eval_closed_loop_nonreactive.sh` and `scripts/eval_closed_loop_reactive.sh`
- README now includes preprocessing commands with `--use-scenario-api` and official closed-loop non-reactive/reactive-IDM instructions.

## Validation performed in this container

- Syntax validation passed:

```bash
python -S -m py_compile $(find dpies -name '*.py')
```

The official nuPlan devkit and full DB/maps are not installed inside this execution container, so real `run_simulation.py` closed-loop execution was not run here. Please run the README closed-loop commands in your nuPlan v1.1 environment and start with `LIMIT_SCENARIOS=1`.

## Remaining engineering caveat

The planner is connected to official closed-loop simulation, but candidate action rollouts are still generated by the repository's lightweight ego-frame rollout generator. For best closed-loop quality, the next major improvement is replacing lane-change/merge/proceed rollouts with fully lane-centerline/lane-connector spline rollouts driven by route topology.
